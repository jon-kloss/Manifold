# DECISIONS

One line per judgment call beyond the docs — what, why, which doc section it touches. Review surface for the human.

- **Workspace layout:** `solver` crate holds both T0 and T1; `solver-wasm` is a thin cdylib wrapper over T0 for wasm-pack (SDD §2 names "solver-core → WASM"; a separate wrapper keeps good_lp/LP deps out of the wasm build). — SDD §2
- **Renderer directory:** React app lives in `renderer/` (not `app/`) to avoid colliding with the `app` crate name. — SDD §2
- **Dev bridge:** `app` crate ships two entry points — the Tauri shell and a `dev-bridge` HTTP+SSE binary exposing the identical command layer, so the renderer can be driven and screenshot in a headless container (no display) against the *real* Rust core. Tauri deps are feature-gated (`shell`, on by default; CI container builds it too now that WebKitGTK is present). Rust still owns canonical state in both modes; no state forking. — SDD §4, HANDOFF "verify as you go"
- **Map imagery:** no community tile set is bundled yet — licensing must be verified before ship (SDD §7). Phase 1 renders the documented placeholder treatment (diagonal-stripe = runtime asset placeholder) over the survey grid + biome ghost labels. **OPEN QUESTION for human: which community tile set / license to pursue.** — SDD §7, UI spec Assets
- **World-node snapshot:** bundled `world-nodes.json` is a hand-assembled subset (iron/copper/limestone/coal in the Grass Fields / Rocky Desert / Northern Forest areas, real-ish coordinates, correct purity distribution shape) sufficient for Phase 1 planning. Full community-data snapshot swap-in is tracked in BACKLOG (v1). — SDD §3 "static world data"
- **Docs.json fixture:** repo bundles a trimmed, UTF-8 Docs.json-format fixture covering the Modular Frame chain (+miners/belts/power) so tests and the dev app run without a game install; the parser detects UTF-16LE (real installs) vs UTF-8 (fixture) by BOM. — SDD §7
- **Internal factory wiring:** SDD §3 models inter-factory `Route`s but not intra-factory connections; the graph view needs them. Added `BeltEdge { factory, from, to, item, tier }` entity (endpoints = machine group or boundary port); flow rates on edges are derived solver output, tier is user-set. Same ◇◈◆ + flow grammar as routes. — SDD §3, UI spec screen 6
- **T0 count semantics:** "scaling counts/clocks/rates linearly" (SDD §5.1) implemented as: required group throughput scales linearly; count = ceil(throughput / machine base rate) with clock redistributed evenly (counts/clocks are numbers, not geometry, so Principle 5 holds). T1 does the same relax-round-redistribute per SDD §5.2.
- **Graph card positions:** added `graph_pos` to `MachineGroup`/`Port` (not in SDD §3) — "solves never move cards" (Principle 5) requires cards to *have* persisted positions; drag commits `move_group_card`/`move_port_card`. — UI spec screen 6
- **Later-phase surfaces omitted, not stubbed:** POWER/DIFF overlay chips, PLAN SUPPLY CHAIN button, advisor edge tab, audit drawer, and route drawing are absent (their subsystems are Phase 2/3/5). FLOWS/NODES keep their spec'd 1/4 key bindings so muscle memory transfers when the others land. — HANDOFF "Later-phase surfaces must not be stubbed"
- **Graph edge encoding on an all-planned factory:** graph belts render the Flow load encoding (color+width+pattern per mock 4a) with *italic* label chips carrying planned-ness; the "planned routes are always blueprint-dashed" rule (1e) applies to map routes, which arrive in Phase 2. Without this, Phase 1's all-planned factories could never show the exit criterion's "belt saturation coloring". — UI spec 1e vs screen 6
- **Alternate recipes render locked** (`NOT UNLOCKED`, 50% opacity) — there is no progression/unlock model until later phases; standard recipes are available. — UI spec screen 6 recipe strip
- **Slider ceiling magnetism:** within 1% of the binding ceiling the target snaps to it, so the hard stop + binding strip engage exactly ("tick at input ceiling — hard-stops there"). — UI spec screen 6
- **Map↔factory transition:** 400ms scale+fade rather than a shared-element zoom morph — keeps the cinematic beat cheap; reduced-motion collapses it globally. Revisit for the true morph in a polish pass (BACKLOG v1.1). — UI spec Motion
- **Claim flow:** claiming a node dispatches `claim_node` + `add_port` (input port with the extraction-rate ceiling) as one undo step; extractor picker offers Miner Mk.1–3. — SDD §3, mock 2a
- **Verification posture:** the Playwright exit-criterion suite drives the real Rust core through the dev-bridge; the Tauri shell compiles in CI but isn't interactively tested in the headless container (no display). First manual desktop smoke test is on the human. — SDD §11
- **Phase 1 exit tag:** annotated tag `phase-1` exists locally on commit `040ebd0` but the GitHub integration rejects tag pushes — the human should re-tag after merge (`git tag -a phase-1 040ebd0`). — HANDOFF commit discipline
- **Belt-style edges (user request):** graph edges now use a layout engine (`edgeLayout.ts`) — anchors spread deterministically along card faces (sorted by counterpart position), orthogonal runs with rounded corners like real conveyors, and schematic *hop* arcs where a belt crosses another. Anchors are visual only; connect-drag still uses the RF handles. — UI spec screen 6 spirit
- **Machine footprints (user request):** cards carry a top-down footprint strip (one shared px-per-meter scale, ≤12 outlines + overflow count) and the inspector shows `w×l m · ×n → total m²`. Dimensions are a community-documented table in `footprints.ts` — Docs.json has none; pak extraction (v2) can replace it. — SDD §7
- **Floors (user request):** `MachineGroup.floor` (serde-default 0, so existing plan files load unchanged) with `set_group_floor` command; context-bar F-chips filter (dim off-floor cards/belts), footer F-badges, inspector stepper; cross-floor belts render as lifts (⇅ on the label). Ports stay on the ground floor. — SDD §3 extension
- **Belt chip declutter (user request):** chips place collision-aware — candidates along the belt (long horizontal runs first), rejecting spots that overlap cards or already placed chips; longest belts place first. Belts shorter than 150 units compact the chip to the saturation % (full text on hover/selection) — a measured deviation from "always-present `n/capacity · %`" (1e): CRIT and selected belts always carry the full chip, so the alarm channel never compacts. — UI spec 1e
- **Floor aesthetics (user request):** floors read through structure, keeping the color law intact — (1) *floor plates*: translucent labeled hulls behind each floor's cards (`FLOOR 1 · 2 GROUPS · 1⤓ 2 IN`), rendered only when ≥2 floors exist; (2) *lift portals*: on a filtered floor, a cross-floor belt runs to a clickable square portal pin (square = infrastructure, A2.3) labeled `⤒/⤓ F<n> · %` that jumps to the connected floor with the belt selected; ghosted off-floor cards become non-interactive; (3) ALL view lift belts carry `⇅ F0→F1` chips and square lift pads at both ends. — A2.3 grammar, SDD §3 extension
- **STACK FLOORS (user request):** context-bar action arranging each floor's cards into its own horizontal band — highest floor on top, cutaway-elevation style — preserving intra-floor layout, snapped to the 16px grid, committed as one undoable step (plain `move_group_card` commands; the solver is untouched since geometry is user-owned, Principle 5).
- **AUTO-FLOOR (user request):** assigns floor = production stage (longest-path topological depth over the intra-factory group DAG; inputs low, final assembly high), then band-stacks — one undoable step of set_group_floor + move_group_card commands. Cyclic graphs bail without touching the plan (no stages to compute; loops arrive with fuel chains in later phases).
- **Belt junctions (user request):** `Junction` entity (splitter / smart splitter / programmable splitter / merger / storage) with the game's physical port budgets enforced on connect (splitter 1-in/3-out, merger 3-in/1-out, storage 1/1); standard junctions carry one item, smart/programmable may filter per output (the edge's item is the filter). Solvers treat junctions as pure per-item conservation nodes (Σin = Σout) — golden test proves they never change the math. Placed via the context bar + LOGISTIC menu; full floors/stack/auto-floor/undo/persistence support. — SDD §3 extension
- **Buildables catalog (user request):** gamedata now parses *every* `FGBuildable*` class from Docs.json into a generic `buildables` table (class, display name, native class) persisted in gamedata.sqlite and served to the renderer — anything the game data names can be displayed; specialized tables (machines/belts/extractors) stay the solver-relevant layer on top.
- **Empire recompute (Phase 2):** every edit runs one `empire_solve` pass — factories topo-sorted over belt routes, each solved with its In-port ceilings tightened to `min(canonical, upstream supply, belt tier cap)`, out-rates fed downstream, route manifests/flows written back into the same undo entry. A cycle in the route graph falls back to stable order and flags `empire_cycle`. Clamp write-back only applies to the port the user explicitly edited — an upstream dip surfaces as a DEFICIT row, never a silent rewrite of a downstream target. — SDD §12, Principle 2
- **Route paths are straight lines (Phase 2):** map routes render pin-to-pin (blueprint-dashed while ◇); the path array is already multi-point in the model, so waypoint editing can land later without a schema change (BACKLOG v1.1). — SDD §3
- **Power as pseudo-item (Phase 2):** `__PowerMW` (1 item/min ≡ 1 MW) with synthesized burn recipes per generator×fuel (`rate = MW·60/energy_MJ`) makes generation flow through the ordinary T0/T1 pipeline — the MW slider back-solves generator count + fuel exactly like items. Supplemental water is deferred with pipes (BACKLOG v1). — A2, SDD §5
- **Circuits are derived, not entities (Phase 2):** grids are recomputed union-find components over power-line routes each solve, named GRID A/B/… by member order; margin grammar OK ≥20% headroom / WARN 5–20 / CRIT <5. Persisted `Circuit` entities (naming, breakers) arrive with priority switches (BACKLOG v1). — A2.1
- **Power lines bind factories, not ports (Phase 2):** `add_route {kind: power}` joins two factory pins (power is a bus, A2.1) — no manifest, no tier; duplicates per pair rejected. The route popover offers "⚡ Power line" alongside belt candidates and never offers the MW pseudo-item as a belt.
- **Audit drawer (Phase 2):** TAB toggles a bottom HUD (46%, pinnable to 82%) with SATURATION (routes + intra belts, TRACE/UPGRADE TIER), DEFICITS (starvation + node conflicts), POWER (circuit margins + factory draws), PLAN DRIFT (empty until save import). Badges: hot belts ≥70%, deficit rows, non-OK circuits. — mock 2d, UI spec 1i
- **Leaflet stacking fix:** `.map-leaflet` gets `z-index: 0` so Leaflet's internal pane z-indexes (200–700) can't bleed over app chrome — found when factory pins rendered through the audit drawer.
- **Phase 2 exit tag:** annotated tag `phase-2` exists locally on the e2e-verified empire commit — like `phase-1`, the integration rejects tag pushes; the human should re-tag after merge. Exit criterion covered by `phase2-empire.spec.ts`: a 5-factory empire (ore → ingots → rods → screws → depot + coal plant) with live route saturation, deficits on upstream dips, and GRID A power margins. — SDD §12
- **Elevation groundwork (user request):** `MapPos` gains `z` (serde-default 0 — old plans load unchanged); world nodes gain `z` + `zone` (surface/cave) + `entrance` for cave nodes. Route length is now 3D and `DerivedRoute` carries climb up/down; factory elevation is planner-entered (SITE section in the summary drawer) since no heightmap is licensed. Pin drags preserve z; `move_factory_pin` refreshes the endpoint waypoints of every bound route so lines and lengths track re-siting. Belts ignore climb in solver math (correct per game); it becomes pump head-lift when pipes land and grade checks when rail lands. — SDD §3 extension
- **Cave nodes (user request):** rendered with an under-arc + `▾CAVE` label; hover/selection reveals the surface entrance (square = infrastructure, A2.3) with a dotted link, and the node drawer explains routing via the entrance. No terrain pathfinding — geometry stays user-owned (Principle 5).
- **Proposal aliases (Phase 3):** items carry the concrete commands they materialize to; cross-item references use `$alias` placeholder ids (Id is a string) resolved to real ULIDs at accept in dependency order — zero command-schema invasion, unresolved aliases abort before anything commits. — SDD §3/§10
- **Toggle cascade (Phase 3):** excluding a review row transitively excludes its dependents; re-including a row pulls its *dependencies* back in but never resurrects dependents — the cascade never guesses what the user wants back. — mock 3a
- **Consequence eval = scratch solve (Phase 3):** partial-accept recompute applies included items to a cloned PlanState, runs the ordinary empire solve, diffs (goal delta, power, new deficits, critical margins), and discards — one code path, no parallel arithmetic. — mock 3a
- **Staleness is derived (Phase 3):** proposals store an FNV-1a hash of the plan projection (proposals excluded, or drafting would self-stale); the STALE badge is `inputHash ≠ live planHash`. RESOLVE = re-solve into a new numbered draft; no stored stale flag to desync. — mock 3c
- **Recipe selection is per-item scoring, not MILP (Phase 3):** candidates scored min-machines-then-min-power independently per item — honest at fixture scale where alternates don't create cross-item trade-offs; true MILP lands when they do (BACKLOG). Alternates default OFF in the wizard (no unlock model; they render locked) and T2 suggests them explicitly, flagged NOT UNLOCKED.
- **A2.4 power sourcing (Phase 3):** the global solver appends generator expansion (SetPortRate on the MW port, sized demand × (1+margin cap)) + a power line when the empire has any generator; with no grid at all it logs the unsourced draw honestly instead of inventing a fuel chain with no claims. Max-new-sites currently caps at the single integrated site the solver produces; multi-site decomposition is backlogged.
- **Priority switches (Phase 3, A2.3):** `PrioritySwitch` sits ON a power route (square pin at its midpoint); shed order is P8→P1. SHEDS AT accumulates: switch s sheds when circuit demand exceeds generation + Σ downstream of earlier sheds; the load side of a cut is the half with less generation. Audit POWER tab gains PRIORITY/SHEDS AT rows + a BROWNOUT SIM row per grid. Deleting a line cascades its switches.
- **Phase 3 exit tag:** annotated tag `phase-3` local only (integration rejects tag pushes; human re-tags after merge). Exit criterion covered by `phase3-proposals.spec.ts`: wizard → reviewable proposal → partial accept with live recompute → accept as ONE undo step → undo restores everything. — SDD §12
- **Transport math (Phase 4, A3):** one `TransportParams` table drives rail/truck/drone; the inspector renders the full math block (travel × 1.12 terrain, dwell, editable 15% headway, RTT, throughput = units × slots × stack ÷ RTT) and the empire recompute uses that computed throughput as the route's capacity — throughput is computed, never asserted (Principle 10). Drones skip the terrain factor (straight-line by design). Signal-block capacity stays v2.
- **Cargo kinds interchange (Phase 4):** belt/rail/truck/drone all bind the same Out→In port pair, so `set_route_spec` swaps them freely (and edits consists/cars/dwell); power/pipe endpoints differ and never interchange. Rail manifests are single-item in v1 — multi-item manifests need multi-port binding (BACKLOG).
- **Import clustering (Phase 4, SDD §8):** DBSCAN eps 120 m, min_pts 1 over manufacturer+generator positions (save cm ÷ 100 → map meters); groups by (machine, recipe) with mean clock; names from the dominant output ("IRON INGOT WORKS 1"). Belt-bridged cluster merging deferred — belts' connection components aren't resolved in v1 (BACKLOG). Dunarr-076: 74k objects → 867 machines → 13 factories, 88 groups, <100 ms cluster+write.
- **Quarantine = non-vanilla paths (Phase 4):** worker quarantines Build_/BP_ classes outside /Game/FactoryGame/ (4,456 SML-modded objects in Dunarr-076), counted in the preview, never silently dropped. Vanilla decor (foundations, walls) is recognized-and-ignored, not quarantined.
- **Re-import accept syncs the Built layer (Phase 4):** drift items carry `SyncOp` payloads instead of commands; accepting a SaveReimport proposal updates ◆ built counts/groups directly — the ONE documented exception to "accept creates ◇ planned only", because the proposal records what already happened in the game. Still one undo entry.
- **Imported factories may not solve (Phase 4):** real saves reference the full recipe catalog; with the bundled fixture gamedata, unknown recipes leave a factory rendered (name, position, ◆ groups, counts) but unsolved (`solve_error`) — honest degradation until a real Docs.json install is pointed at (`FICSIT_DOCS_JSON`). Import stays enrichment, never load-bearing (Principle 6).
- **Phase 4 exit tag:** annotated tag `phase-4` local only (human re-tags after merge). Exit covered by `phase4-import.spec.ts`: Dunarr-076 imports as the built layer through the real worker+UI, one undo removes it, identical re-import reports IN SYNC, a divergent save renders drift rows in the DIFF (PLAN DRIFT) tab with REVIEW into the SaveReimport proposal — which is never auto-applied. — SDD §12
- **Advisor gating (Phase 5, SDD §9):** cards fire only when a condition BECOMES true (edge-detect on condition keys, not rules), at most once per rule per 30 s, never for a muted rule, never while paused. Dismiss = mute that rule, persisted — the anti-nag contract is "dismissing means stop telling me about this". Cards/mutes persist OUTSIDE the undo journal (they record what the advisor saw; undoing a plan edit must not eat them). Rules: node_conflict, new_deficit, saturation_high (≥75%), power_swing (<20% headroom), drift_detected (open SaveReimport proposal). Every CTA routes through existing review surfaces — the advisor never edits the plan.
- **AI offline is a first-class mode (Phase 5):** no `FICSIT_AI_KEY` → `AI OFFLINE` chip and the SAME heuristic feed; the hourly model-call budget (6/h) is displayed but unspent. Chat runs a deterministic engine offline: status questions answer from derived state with SAW provenance, and "produce <item> at <rate>/min" intents materialize through the SAME global-solver validation path a model's `proposal_intent` would — the model never mutates state either way, so offline and online differ only in prose. The streaming model relay is scaffolded behind the key check (BACKLOG).
- **Context snapshots are user-viewable (Phase 5):** `compact_state(scope)` is exactly what a model would receive — aggregated rollups, top-10 deficits, circuits — with the serialized byte size shown in the chat context bar and the full JSON behind ▸ VIEW. Empire snapshots stay well under the 30 KB target at current scale.
- **Style guides (Phase 5):** typed `StyleGuide` entity (palette/massing/techniques/sequence + source note) with create/delete/set-theme commands; factories link via the existing `styleGuide` field. The image→guide vision call is offline-gated: + NEW GUIDE creates an honest manual template noting "aesthetic inference needs a model key".
- **Icon pack stays placeholder (Phase 5):** `AssetSource` trait (Install/CommunityPack/Placeholder) landed with `PlaceholderAssets` only — the pinned community icon pack shares the SAME unresolved licensing question as the map tiles (human decision). The diagonal-stripe chip remains a first-class rendering, never an error state. — SDD §7
- **Onboarding is one card (Phase 5, mock 7c):** steps 1–2's install-path detection and icon-extraction theater don't apply while the catalog is the bundled fixture — the card states the active catalog source honestly (`FICSIT_DOCS_JSON` to re-extract) and offers the three doors (N/P/S). No tour; dismiss persists per plan.
- **Phase 5 exit tag:** `phase-5` local only (human re-tags after merge). The "survives a week of dogfooding without feeling naggy" criterion is qualitative — the mechanical guarantees behind it (edge-arming, debounce, persistent mutes, pause, badge-count-only presence) are unit-tested and e2e-verified in `phase5-advisor.spec.ts`; the week of dogfooding is yours. — SDD §12
