# DECISIONS

One line per judgment call beyond the docs — what, why, which doc section it touches. Review surface for the human.

- **Workspace layout:** `solver` crate holds both T0 and T1; `solver-wasm` is a thin cdylib wrapper over T0 for wasm-pack (SDD §2 names "solver-core → WASM"; a separate wrapper keeps good_lp/LP deps out of the wasm build). — SDD §2
- **Renderer directory:** React app lives in `renderer/` (not `app/`) to avoid colliding with the `app` crate name. — SDD §2
- **Dev bridge:** `app` crate ships two entry points — the Tauri shell and a `dev-bridge` HTTP+SSE binary exposing the identical command layer, so the renderer can be driven and screenshot in a headless container (no display) against the *real* Rust core. Tauri deps are feature-gated (`shell`, on by default; CI container builds it too now that WebKitGTK is present). Rust still owns canonical state in both modes; no state forking. — SDD §4, HANDOFF "verify as you go"
- **Map imagery:** no community tile set is bundled yet — licensing must be verified before ship (SDD §7). Phase 1 renders the documented placeholder treatment (diagonal-stripe = runtime asset placeholder) over the survey grid + biome ghost labels. **OPEN QUESTION for human: which community tile set / license to pursue.** — SDD §7, UI spec Assets
- **World-node snapshot:** bundled `world-nodes.json` is a hand-assembled subset (iron/copper/limestone/coal in the Grass Fields / Rocky Desert / Northern Forest areas, real-ish coordinates, correct purity distribution shape) sufficient for Phase 1 planning. Full community-data snapshot swap-in is tracked in BACKLOG (v1). — SDD §3 "static world data"
- **Docs.json fixture:** repo bundles a trimmed, UTF-8 Docs.json-format fixture covering the Modular Frame chain (+miners/belts/power) so tests and the dev app run without a game install; the parser detects UTF-16LE (real installs) vs UTF-8 (fixture) by BOM. — SDD §7
- **Internal factory wiring:** SDD §3 models inter-factory `Route`s but not intra-factory connections; the graph view needs them. Added `BeltEdge { factory, from, to, item, tier }` entity (endpoints = machine group or boundary port); flow rates on edges are derived solver output, tier is user-set. Same ◇◈◆ + flow grammar as routes. — SDD §3, UI spec screen 6
- **T0 count semantics:** "scaling counts/clocks/rates linearly" (SDD §5.1) implemented as: required group throughput scales linearly; count = ceil(throughput / machine base rate) with clock redistributed evenly (counts/clocks are numbers, not geometry, so Principle 5 holds). T1 does the same relax-round-redistribute per SDD §5.2.
- **Graph card positions:** added `graph_pos` to `MachineGroup`/`Port` (not in SDD §3) — "solves never move cards" (Principle 5) requires cards to *have* persisted positions; drag commits `move_group_card`/`move_port_card`. — UI spec screen 6
- **Later-phase surfaces omitted, not stubbed:** POWER/DIFF overlay chips, PLAN SUPPLY CHAIN button, advisor edge tab, audit drawer, and route drawing are absent (their subsystems are Phase 2/3/5). FLOWS/NODES keep their spec'd 1/4 key bindings so muscle memory transfers when the others land. — HANDOFF "Later-phase surfaces must not be stubbed"

Without this, Phase 1's all-planned factories could never show the exit criterion's "belt saturation coloring". — UI spec 1e vs screen 6
- **Alternate recipes render locked** (`NOT UNLOCKED`, 50% opacity) — there is no progression/unlock model until later phases; standard recipes are available. — UI spec screen 6 recipe strip
- **Slider ceiling magnetism:** within 1% of the binding ceiling the target snaps to it, so the hard stop + binding strip engage exactly ("tick at input ceiling — hard-stops there"). — UI spec screen 6
- **Map↔factory transition:** 400ms scale+fade rather than a shared-element zoom morph — keeps the cinematic beat cheap; reduced-motion collapses it globally. Revisit for the true morph in a polish pass (BACKLOG v1.1). — UI spec Motion

- **Verification posture:** the Playwright exit-criterion suite drives the real Rust core through the dev-bridge; the Tauri shell compiles in CI but isn't interactively tested in the headless container (no display). First manual desktop smoke test is on the human. — SDD §11

Cyclic graphs bail without touching the plan (no stages to compute; loops arrive with fuel chains in later phases).
- **Belt junctions (user request):** `Junction` entity (splitter / smart splitter / programmable splitter / merger / storage) with the game's physical port budgets enforced on connect (splitter 1-in/3-out, merger 3-in/1-out, storage 1/1); standard junctions carry one item, smart/programmable may filter per output (the edge's item is the filter). - **Empire recompute (Phase 2):** every edit runs one `empire_solve` pass — factories topo-sorted over belt routes, each solved with its In-port ceilings tightened to `min(canonical, upstream supply, belt tier cap)`, out-rates fed downstream, route manifests/flows written back into the same undo entry. A cycle in the route graph falls back to stable order and flags `empire_cycle`. - **A1 hard floor removed (post-review, user call):** the 1366×768 refuse screen measured CSS pixels, which locked out high-DPI displays whose *logical* resolution is small (4K TV at 300% OS scaling ≈ 1088×693). There is no minimum screen size anymore: below 1600 px everything degrades to the overlay layout, and the Tauri shell auto-zooms the webview out (to at most 60%) so the layout regains room while text stays physically large (`useAutoZoom`). The refuse card and `LayoutMode::refuse` are gone; error/hydrating screens keep the titlebar so the frameless window always has drag + min/max/close. — supersedes A1's "refuse gracefully" clause
- **Imported factories are auto-wired logically (user report, post-v0.1):** the save parser extracts machines but not belt connectivity, and an unwired factory solves as nothing — so `write_built_layer` wires each cluster by recipe I/O: producer groups edge into consumer groups per item, net consumption becomes In ports (surfacing as honest deficits until routed), net surplus becomes Out ports. This is logical structure, not the game's physical belt runs. Corollary invariant enforced in the same change: **the solver never resizes ◆ built groups** — T1 write-back skips `Status::Built` (import sync is the only writer of the built layer).
- **Layered graph auto-layout (user feedback, post-v0.1):** generated factories were grid-dumped and edge crossings made flow unreadable. New `planner-core::layout` — Sugiyama-lite: In ports column 0, groups/junctions ranked by flow depth (longest-path, cycle-tolerant), Out ports last, barycenter sweeps minimize crossings, rows stacked by real card heights. Imports lay out at creation; `TidyLayout` command (TIDY in the graph context bar) re-lays any factory as one undoable, cosmetic-only step — it applies to built cards too since positions aren't game truth.
- **Claim tethers on the world map (user feedback, post-v0.1):** node↔factory assignment was invisible outside the node drawer. Claimed nodes now tether to their owning pin: quiet dashed steel line; blueprint-dashed when the claim is ◇ planned; crit red when conflicted; selecting/hovering the factory or node promotes the tether to signal orange and names the factory at the midpoint ("orange is a verb" — highlight follows selection, never resting state). Legend row added.
- **Built-edit deltas (SDD §3.1.1, code-review M2):** `planned_delta` is realized as an inline `GroupDelta { count: Option<u32>, clock: Option<f64> }` — an overlay of planned *effective* values on the ◆ built baseline — not the SDD's literal `DeltaRef` entity (a referenced ◇ group composes only for count increases, would render as a phantom second card, and needs a new collection; the overlay handles increase, decrease, and clock retune uniformly, and undo comes free from the group's existing patch pair). Per-component `Option` so a clock-only retune never pins the count: unedited components keep tracking the baseline through re-import sync. `SetGroupCount`/`SetGroupClock` on ◆ write the overlay (setting the built value back clears that component; an empty overlay is `None`); baseline count/clock stay import-owned ground truth. The solver READS effective values but never WRITES deltas — deltas are user intent, materialized only by explicit commands; note the deliberate asymmetry: T1 redistributes counts/clocks for ◇ planned groups but leaves built-group deltas literal (a target drag never sprays ◇ deltas across an imported factory; solver-generated deltas are BACKLOG). Re-import sync writes the baseline and KEEPS the delta, clearing any component the game caught up to — the Inspector's "visible until built in-game" dissolves automatically. `SetGroupRecipe`/`SetGroupFloor`/`DeleteGroup` on ◆ stay rejected: those invalidate the baseline rather than overlay it (structural/demolition deltas are BACKLOG v1.1). No plan-file migration: every existing file serialized `plannedDelta: null`. — SDD §3.1.1
- **Polish pass batch 1 (user-driven, post-review):** map label declutter — pin chips, route/grid/switch label chips, and node ghost labels collision-cull deterministically (selected wins, hover reveals a culled pin chip; lines/pins always draw); parallel routes between the same factory pair fan into stable 9-px screen-space lanes shared by draw and hit-test (cargo endpoints normalize port→factory so mixed kinds share the fan); boundary In ports state their real source (FROM ROUTE / FROM NODE CLAIM / UNROUTED — SUPPLY ASSUMED — the old label lied on imported factories); backend error chips resolve ULIDs to display names; F-framing pads for the open inspector; audit rows name their destination; import preview's quarantine list is viewable.
- **Real world map + real nodes (user request):** the placeholder 12-node snapshot is replaced by the full vanilla dataset — 459 solid/oil nodes generated from a vendored MIT community extraction (`scripts/gen-world-nodes.py`, provenance in NOTICE.md) — and the map home renders the real in-game terrain under the data layer by default (TERRAIN chip / key 3 to toggle). The terrain is NOT a Leaflet pane: it's blitted as the bottom layer of the existing data canvas (pre-filtered once — saturate .5 / brightness .55 — so the data layer keeps owning color), because the canvas sits under the map-pane stacking context (pins above canvas) and any opaque pane content would occlude every canvas-drawn node/grid/tether. Same-canvas drawing also keeps terrain and data in lockstep on every pan/zoom frame. Fracking satellites, geysers, and cave/entrance zoning aren't in the dataset (BACKLOG); the schema and cave rendering stay.
- **Built-immutability matrix (code-review, planner-core-integrity):** tiers are physical game infrastructure, so `SetEdgeTier`/`SetRouteTier` now `require_planned` (consistent with DeleteEdge and SetRouteSpec — imported ◆ belts are corrected by re-import sync, not hand edits; tier-upgrade-as-planned-delta is BACKLOG alongside structural group deltas). `RenameFactory` stays deliberately unguarded: names are planner-side labels the save format doesn't know — import synthesizes them and re-import matching is positional, same reasoning as TidyLayout/card moves on ◆. `SetPortRate`/`SetPortCeiling` stay unguarded: targets/ceilings are solver intent, not save-file truth — guarding them would dead-end every imported factory. `ReleaseNode` unguarded is an acknowledged deferral until import creates ◆ claims. — SDD §3.1.1
- **Flow notation on the world map (user request, part 2):** route lines now carry map notation so kind + direction survive label culling — flow chevrons on every cargo route (path order = from → to), crossties on rail, cargo squares on trucks, dotted lines for drones; power lines stay glyph-free (a bus, not a flow). Label chips degrade instead of vanishing: when the full item·flow/cap·sat% chip can't fit, a tag-only micro-chip (MK.2 / RAIL / …) tries first — "level of each" stays readable in dense views. Node dots scale with zoom (3.5px at world view → 7px zoomed in; ghost labels drop at world zoom) so the 459-node field reads as background under factories and flows. Unknown item classes prettify (Desc_SteelIngot_C → STEEL INGOT) rather than leaking raw class names into chips.
- **Build queue is derived (W1c):** the "what do I do next" queue is a pure projection over canonical state (`derive_build_queue(state, gamedata)` in `crates/app/src/buildqueue.rs`), recomputed every empire solve like circuits/deficits — no ordering entity, no stored done-flag. A step is an existing ◇ planned factory (a rollup over its groups), a ◆ built group carrying a planned delta, or a planned route/claim; order is `(proposal number, entity ULID)` with manual entities in a MANUAL bucket (0). Completion is DERIVED through a shared `nearest_built_match` (250m, the same rule as re-import drift diffing, so "is it built yet" and "did the save move it" never disagree): a planned factory is ◆ Done when a built twin covers its (machine,recipe) groups, ◈ Partial if only some — ◈ is a rollup, NEVER written to `status` (that would trip `require_planned` and violate "◆ is game ground truth"). `accept_proposal` now stamps `CreatedBy::Proposal(pid)` on the ◇ entities it materializes (the raw commands default to Manual) so the queue buckets by proposal number and lights milestone progress; ◆ entities from drift sync keep their Import provenance. Milestone "built" is gross ◆ production of the item (the save has no lifetime-crafted counter). — SDD §8, mock 2d
- **Build override overlay (W1c):** the ONE new canonical, un-derivable, undoable state — a serde-default `build_overrides: BTreeMap<Id, BuildOverride>` collection + `Entity::BuildOverride` wired exactly like switches/styleGuides (+ `COLL_BUILD_OVERRIDES`), mutated by `SetBuildDone { id, done: Option<bool> }` through `state.upsert`/`remove` (Some → pin, None → clear) for one free undo entry; no `require_planned` (it's build-progress metadata, not a ◆ mutation). Resolution is `done = override ?? derived`.
